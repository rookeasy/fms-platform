# Project Atlas Day Light UI

## Purpose

Project Atlas Day evolves the Fuzion Operating Platform into a calm, professional, light operating environment for daily work.

The application UI should feel engineering-led, spacious, and durable. Darker brand treatment remains appropriate for login, front door, and marketing surfaces.

## Palette

- Primary navy: `#0F172A`
- Deep navy: `#111827`
- Warm white background: `#FAFAF8`
- Surface white: `#FFFFFF`
- Soft panel: `#F8FAFC`
- Border: `#E2E8F0`
- Slate text: `#334155`
- Muted text: `#64748B`
- FOP coral: `#F15B4A`
- Coral soft background: `#FFF1EE`
- Success: `#16A34A`
- Warning: `#D97706`
- Critical: `#DC2626`

## UI Usage Rules

- Use warm white for the app background.
- Use white cards for primary work surfaces.
- Use soft panels for grouped secondary content.
- Use subtle slate borders instead of heavy dark separation.
- Use primary navy for headings and key values.
- Use slate text for body copy and muted labels.
- Keep the sidebar deep navy as the persistent product identity area.
- Keep the top header quiet: page title, search, notifications, and user menu.

## Atlas Day vs Atlas Night

Atlas Day is for the authenticated operating system: dashboards, registries, properties, buildings, closeout, work orders, inspections, and administration.

Atlas Night is reserved for branded entry points: login, public front door, loading/splash moments, and marketing-adjacent pages where stronger atmosphere is useful.

## Coral Usage Principle

Coral is an indicator, not a background theme.

Use coral for:

- Primary actions.
- Active navigation.
- Protected or active state emphasis.
- Focus rings and important progress accents.

Avoid using coral for large page backgrounds, decorative blobs, or repeated ornamental elements.

## Future Guidance

- Prefer fewer competing metrics per screen.
- Keep each page centered on one primary message.
- Let cards and tables use consistent spacing, borders, and type hierarchy.
- Avoid black-heavy surfaces inside the authenticated app.
- Preserve FOP branding in user-facing UI without renaming backend modules, routes, schemas, or database objects.
